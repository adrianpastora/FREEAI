# API reference

> Interactive docs are always available at `http://localhost:8000/docs`
> (Swagger UI, generated by FastAPI). This document is the opinionated version
> — it explains not just the shapes, but *why* the endpoints are the way they
> are and *which* ones you actually need.

## Auth overview

Two independent auth layers. Which one applies depends on the route prefix.

| Prefix | Credential | Header | How it's checked |
|---|---|---|---|
| `/v1/*` | **Client API key** (`fai_…`) issued from `POST /api/clients` | `Authorization: Bearer fai_…` | SHA-256 hash lookup in `clients` table |
| `/api/*` | **Admin token** printed at startup (or `FREEAI_ADMIN_TOKEN`) | `X-Admin-Token: adm_…` or `Authorization: Bearer adm_…` | `secrets.compare_digest` against the server's static token |
| `/api/health`, `/metrics`, static frontend | — | — | public |

### Bootstrap mode

If the `clients` table is empty **and** `FREEAI_REQUIRE_AUTH=false` (the
default), `/v1/*` is open — useful for first-run experimentation. The app logs
a `bootstrap_mode` warning at startup and `/api/health` returns
`auth_required: false`.

The moment you create your first client, or set `FREEAI_REQUIRE_AUTH=true`,
`/v1/*` requires a Bearer key forever.

## Error format

All errors follow one of two shapes:

```json
{"detail": "plain string"}
```

for validation errors and simple 4xx, or:

```json
{
  "detail": {
    "provider": "groq",
    "kind": "rate_limited",
    "message": "Rate limit exceeded"
  }
}
```

for provider-level errors where the `kind` is the
[`ErrorKind`](../backend/app/providers/base.py) enum value. Client code should
branch on `kind`, not on the message string.

HTTP status mapping (from `main.py::_KIND_TO_STATUS`):

| `kind` | HTTP status |
|---|---|
| `client_error` | 400 |
| `auth`, `server_error`, `parsing`, `unknown` | 502 |
| `rate_limited` | 503 |
| `network` | 504 |

Every response carries `X-Request-ID: <hex>` — include it when reporting bugs.

## Chat completions

### `POST /v1/chat/completions`

**Auth:** client (or bootstrap)

OpenAI-compatible. Accepts the standard chat payload plus FreeAI extras.

```json
{
  "messages": [
    {"role": "system", "content": "You are terse."},
    {"role": "user", "content": "Why is the sky blue?"}
  ],
  "model": null,
  "strategy": "auto",
  "temperature": 0.7,
  "max_tokens": 512,
  "stream": false,
  "preferred_provider": null,
  "fallback": true
}
```

| Field | Type | Notes |
|---|---|---|
| `messages` | `[{role, content, name?}]` | Required. `role ∈ {system, user, assistant, tool}`. |
| `model` | `string \| null` | Leave `null` to use the provider's `default_model`. If set, it's passed through as-is to whichever provider the router picks — careful mixing `model` with `preferred_provider`. |
| `strategy` | `string` | Any strategy name from `GET /api/strategies`. `auto` runs the detector. |
| `temperature` | `float` | Passed to the provider. Default 0.7. |
| `max_tokens` | `int \| null` | Passed to the provider if set. |
| `stream` | `bool` | If `true`, returns SSE (see below). |
| `preferred_provider` | `string \| null` | Force a specific provider. Still respects rate limits / health; if unavailable, the request still falls back. |
| `fallback` | `bool` | Set `false` to get a single attempt, no chain. |

**Non-streaming response:**

```json
{
  "id": "freeai-a1b2c3d4e5f6",
  "object": "chat.completion",
  "created": 1712840000,
  "model": "llama-3.3-70b-versatile",
  "provider": "groq",
  "strategy_used": "coding",
  "choices": [
    {"index": 0, "message": {"role": "assistant", "content": "..."}, "finish_reason": "stop"}
  ],
  "usage": {"prompt_tokens": 42, "completion_tokens": 128, "total_tokens": 170},
  "latency_ms": 487,
  "fallback_chain": ["groq"]
}
```

The FreeAI-only fields (`provider`, `strategy_used`, `latency_ms`, `fallback_chain`) don't
exist in the OpenAI spec but are safe for any reasonable client — they're just extra keys.

**Streaming response:**

Set `stream: true`, get back `Content-Type: text/event-stream`:

```
data: {"id":"freeai-…","choices":[{"delta":{"role":"assistant","content":"He"}}],"provider":"groq",…}

data: {"id":"freeai-…","choices":[{"delta":{"role":"assistant","content":"llo"}}],"provider":"groq",…}

data: {"id":"freeai-…","choices":[{"delta":{},"finish_reason":"stop"}],"provider":"groq",…}

data: [DONE]
```

The shape of each chunk mirrors OpenAI's streaming format. FreeAI adds
`provider`, `strategy_used` on every chunk so the UI can show them from the
first frame without waiting for the end.

**Streaming and fallback:** mid-stream fallback is impossible — once a chunk
is sent we can't switch providers. If the first request fails *before* the
first chunk, the orchestrator falls back. If it fails after, the stream is
terminated with an error event:

```
data: {"error":{"provider":"groq","kind":"server_error","message":"…"}}
data: [DONE]
```

### Drop-in OpenAI SDK example

```python
from openai import OpenAI
client = OpenAI(base_url="http://localhost:8000/v1", api_key="fai_…")

# non-streaming
resp = client.chat.completions.create(
    model="auto",
    messages=[{"role": "user", "content": "hola"}],
    extra_body={"strategy": "coding"},
)

# streaming
for chunk in client.chat.completions.create(
    model="auto",
    messages=[{"role": "user", "content": "explain TLS"}],
    stream=True,
    extra_body={"strategy": "reasoning"},
):
    print(chunk.choices[0].delta.content or "", end="", flush=True)
```

## Admin — providers

### `GET /api/providers`

**Auth:** admin

Returns live status for every registered provider:

```json
[
  {
    "name": "groq",
    "enabled": true,
    "has_key": true,
    "healthy": true,
    "requests_today": 42,
    "requests_this_minute": 3,
    "rpm_limit": 30,
    "rpd_limit": 14400,
    "last_error": null,
    "last_latency_ms": 420,
    "tags": ["fast", "cheap", "coding", "reasoning"],
    "default_model": "llama-3.3-70b-versatile"
  }
]
```

### `PATCH /api/providers/{name}`

**Auth:** admin

Updates any subset of provider config. Any field omitted is left unchanged.
Editing **any** field also resets health (so a provider in quarantine gets a
second chance after you rotate its key).

```json
{
  "api_key": "sk-…",           // pass "" to clear the key
  "enabled": true,
  "weight": 1.5,
  "default_model": "llama-3.1-70b-versatile",
  "rpm_limit": 30,
  "rpd_limit": 14400,
  "tags": ["fast", "cheap"]
}
```

**Response:**

```json
{
  "provider": { /* ProviderStatus shape from GET */ },
  "model_warning": "'llama-3.1-70b-versatile' is not in the known-models list for groq. It may still work — FreeAI will pass it through to the provider.",
  "model_suggestions": ["llama-3.3-70b-versatile"]
}
```

Model validation is **soft** — unknown models still go through. If the model
you typed is in the known list, `model_warning` and `model_suggestions` are
absent / empty.

### `POST /api/providers/{name}/reset`

**Auth:** admin

Force-clears the quarantine flag on a provider. Useful when a 5xx storm
tripped the streak and you know the upstream is back.

```json
{"ok": true}
```

### `GET /api/providers/{name}/models`

**Auth:** admin

Returns the static list of known models for the provider:

```json
{
  "provider": "groq",
  "models": [
    {
      "id": "llama-3.3-70b-versatile",
      "context_window": 128000,
      "capabilities": ["chat", "tools"],
      "note": ""
    }
  ]
}
```

Source: [known_models.py](../backend/app/providers/known_models.py). Not live
from the upstream's `/models` endpoint — it's curated so the UI has something
stable to show.

## Admin — strategies

### `GET /api/strategies`

**Auth:** admin

```json
[
  {
    "name": "coding",
    "tags": ["coding", "reasoning"],
    "description": "Tuned for code, tracebacks, refactors.",
    "is_builtin": true
  },
  {
    "name": "my_rag",
    "tags": ["rag", "quality"],
    "description": "Prefer RAG-tuned models with good reasoning",
    "is_builtin": false
  }
]
```

### `POST /api/strategies`

**Auth:** admin — 409 if a strategy with that name already exists.

```json
{
  "name": "my_rag",
  "tags": ["rag", "quality"],
  "description": "Prefer RAG-tuned models with good reasoning"
}
```

Name must match `^[a-z0-9_]+$` and be ≤32 chars. Custom strategies are never
`is_builtin`.

### `PATCH /api/strategies/{name}`

**Auth:** admin

Edit tags / description of any strategy (custom or built-in). The `is_builtin`
flag is preserved — you cannot flip a built-in to custom or vice versa.

```json
{"name": "coding", "tags": ["coding"], "description": "coder only"}
```

The `name` in the body must match the URL; mismatches return 400.

### `DELETE /api/strategies/{name}`

**Auth:** admin

Delete a custom strategy. Attempting to delete a built-in returns 400.

## Admin — config

### `GET /api/config`

**Auth:** admin

```json
{
  "default_strategy": "auto",
  "enable_fallback": true,
  "available_strategies": ["auto", "cheapest", "coding", "fastest", …],
  "available_providers": ["groq", "gemini", "mistral", …]
}
```

### `PUT /api/config/strategy`

**Auth:** admin — 400 if the strategy doesn't exist.

```json
{"default_strategy": "coding"}
```

This is the strategy used when a client doesn't pass `strategy` in a request.
Individual requests can always override.

### `PUT /api/config/fallback`

**Auth:** admin

```json
{"enable_fallback": false}
```

Kill switch for the whole fallback chain. Use with care — a single provider's
downtime will surface as 5xx to clients until you flip it back.

## Admin — analytics

### `GET /api/analytics`

**Auth:** admin

Query params:
- `window_seconds` — integer, 60..604800 (7 days). Default 86400 (24h).
- `bucket_count` — integer, 1..168. Default 24.

Response:

```json
{
  "window_seconds": 86400,
  "total_calls": 1234,
  "success_calls": 1200,
  "failed_calls": 34,
  "success_rate": 0.9725,
  "p50_latency_ms": 480,
  "p95_latency_ms": 2100,
  "total_tokens": 189340,
  "by_provider": [
    {"provider": "groq", "calls": 900, "success": 890, "avg_latency_ms": 410, "tokens": 140000}
  ],
  "by_strategy": [{"strategy": "auto", "calls": 800}],
  "by_outcome": [{"outcome": "success", "calls": 1200}],
  "time_buckets": [
    {"bucket_start": 1712840000.0, "calls": 42, "success": 40}
  ]
}
```

All aggregates are computed from the `usage_events` table at query time
(`PERCENTILE_CONT` for latency, `FILTER` clauses for success counts). They
include both streaming and non-streaming requests. Note that **streaming
requests don't report token counts** — the providers don't send usage in SSE
chunks, so `total_tokens` underestimates real usage roughly in proportion to
your streaming share. See [REVIEW.md § 4](REVIEW.md#4-streaming-telemetry-gaps).

## Admin — clients

### `GET /api/clients`

**Auth:** admin

```json
[
  {"name": "web-app", "key_hash": "ab12…", "rpm_limit": 60, "enabled": true}
]
```

Notice: raw keys are **never** returned. Only the SHA-256 hash.

### `POST /api/clients`

**Auth:** admin

```json
{"name": "web-app", "rpm_limit": 60}
```

Response (the only time you'll see the raw key):

```json
{
  "name": "web-app",
  "api_key": "fai_DoNotLoseThis_abc123…",
  "key_hash": "ab12…",
  "rpm_limit": 60
}
```

Store it immediately. If the user loses it, revoke and create a new one —
there's no recovery path by design.

### `DELETE /api/clients/{key_hash}`

**Auth:** admin

Revokes the client. Apps using it get 401 on the next request. 404 if the hash
doesn't exist.

## Public

### `GET /api/health`

```json
{
  "status": "ok",
  "providers_configured": 3,
  "clients_configured": 2,
  "auth_required": true
}
```

`auth_required` is the canonical flag for bootstrap mode — the frontend reads
it to decide whether to show the admin-token modal immediately.

### `GET /metrics`

Prometheus exposition format. No auth — put it behind a private network or a
reverse-proxy rule if you're worried. The metrics exposed are listed in
[ARCHITECTURE.md § 5](ARCHITECTURE.md#5-request-observability).

## What the frontend actually uses

If you're integrating a different UI, the endpoint set you need is:

- **Bootstrap:** `GET /api/health` (to detect bootstrap mode)
- **Auth gate:** `GET /api/providers` (401 → show modal)
- **Providers tab:** `GET/PATCH /api/providers`, `POST /api/providers/{name}/reset`
- **Strategy tab:** `GET /api/config`, `GET/POST/PATCH/DELETE /api/strategies`, `PUT /api/config/strategy`, `PUT /api/config/fallback`
- **Clients tab:** `GET/POST/DELETE /api/clients`
- **Analytics tab:** `GET /api/analytics?window_seconds=X&bucket_count=Y`
- **Playground:** `POST /v1/chat/completions` with or without `stream`

That's 13 distinct endpoint methods. Not huge — the interactive docs at
`/docs` are the authoritative spec.
