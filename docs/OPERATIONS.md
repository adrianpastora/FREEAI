# Operations

> How to run, observe, back up and recover FreeAI in production.

## 1. Deployment modes

### 1.1 Docker Compose ŌĆö minimum

```bash
docker compose up --build
```

Two containers:

- `postgres:16-alpine` ŌĆö volume `freeai_pg` persisted on the host
- `freeai` ŌĆö built from [backend/Dockerfile](../backend/Dockerfile), runs
  `uvicorn --workers 2`, healthchecks against `/api/health`

Exposed host ports: **`8000`** (freeai) and **`127.0.0.1:5444`** (postgres).
Postgres is bound to the loopback interface so it never leaks onto a public
network ŌĆö connect from the host with `docker compose exec postgres psql -U
freeai -d freeai` or by pointing a local client at `127.0.0.1:5444`.

**First run (UI)**: if you do **not** set `FREEAI_ADMIN_TOKEN` and there is no
`data/admin_token` file, and no provider API keys are stored yet, the web UI
shows a **first-time setup** modal (same style as the admin lock dialog). You
choose an admin token (stored as a **bcrypt hash** in `app_config`) and can
paste provider keys (stored **encrypted** like normal PATCH /api/providers).

**First run (env / file)**: set the admin token before starting ŌĆö either
`FREEAI_ADMIN_TOKEN` or a `data/admin_token` file (Docker volume). Example:

```bash
FREEAI_ADMIN_TOKEN="adm_$(openssl rand -base64 24 | tr -d '/+=')" \
  docker compose up --build
```

### 1.2 Docker Compose ŌĆö with observability

```bash
docker compose --profile observability up --build
```

Adds two more containers: `prometheus:v2.54.1` and `grafana:11.2.2`. Grafana
is provisioned with a Prometheus datasource and the **FreeAI ŌĆö Overview**
dashboard. See [┬¦ 3](#3-observability) for details.

Default Grafana creds are `admin / admin`. Override with
`GRAFANA_USER` / `GRAFANA_PASSWORD` in the environment.

### 1.3 Bare metal (no Docker)

```bash
# Postgres must already be running
createuser -P freeai     # password: freeai (or whatever)
createdb -O freeai freeai

cd backend
pip install -r requirements.txt

export FREEAI_DATABASE_URL="postgresql+asyncpg://freeai:freeai@localhost:5432/freeai"
export FREEAI_MASTER_KEY="$(python -c 'from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())')"
export FREEAI_ADMIN_TOKEN="adm_$(openssl rand -hex 20)"

# Migrations happen automatically at startup by default
python run.py
```

Or with explicit workers:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

Multi-worker safety is why Sprint 2 existed ŌĆö Postgres holds all state, so
scaling workers is just a knob.

### 1.4 Kubernetes

There's no Helm chart yet; here's the opinionated blueprint:

- `freeai` Deployment with 2+ replicas, `uvicorn --workers 1` per pod (let K8s
  do the replica counting, not uvicorn).
- `postgres` via a managed service (RDS, Cloud SQL, Supabase) ŌĆö not in-cluster
  unless you know your operator.
- Secret with `FREEAI_DATABASE_URL`, `FREEAI_MASTER_KEY`, `FREEAI_ADMIN_TOKEN`.
- `FREEAI_AUTO_MIGRATE=false` on the Deployment, run migrations as a Job
  in a predeploy hook (`alembic upgrade head`).
- `/api/health` as liveness + readiness probe.
- `/metrics` scraped by your Prometheus operator with the same scrape config
  as [deploy/prometheus/prometheus.yml](../deploy/prometheus/prometheus.yml).

## 2. Environment variables

Full list. All prefixed with `FREEAI_` except provider keys. Parsed by
[backend/app/settings.py](../backend/app/settings.py).

| Variable | Default | Purpose |
|---|---|---|
| `FREEAI_DATABASE_URL` | `postgresql+asyncpg://freeai:freeai@localhost:5432/freeai` | async SQLAlchemy URL. **Must use the `asyncpg` driver.** En Compose el servicio `freeai` usa `...@postgres:5432/...` (red interna). |
| `FREEAI_DB_ECHO` | `false` | Log every SQL statement (dev only ŌĆö very noisy) |
| `FREEAI_DB_POOL_SIZE` | `10` | Engine pool size per worker |
| `FREEAI_DB_MAX_OVERFLOW` | `20` | Overflow slots |
| `FREEAI_AUTO_MIGRATE` | `true` | Run `alembic upgrade head` at startup |
| `FREEAI_ADMIN_TOKEN` | autogenerated | Static token for `/api/*`. Autogen'd to `data/admin_token` (chmod 600) if missing. |
| `FREEAI_MASTER_KEY` | autogenerated | Fernet master key for provider API key encryption. Autogen'd to `data/.master_key` (chmod 600) if missing. |
| `FREEAI_REQUIRE_AUTH` | `false` | Force `/v1/*` auth even with zero clients. Disables bootstrap mode. |
| `FREEAI_CORS_ORIGINS` | `http://localhost:8000,http://127.0.0.1:8000` | Comma-separated list |
| `FREEAI_LOG_LEVEL` | `INFO` | Root log level |
| `FREEAI_LOG_FORMAT` | `console` | `console` (dev, colored) or `json` (prod, for log aggregators) |
| `GROQ_API_KEY` | ŌĆö | Seeds the `providers.groq.api_key` column on first startup if empty |
| `GEMINI_API_KEY` | ŌĆö | Same for gemini |
| `MISTRAL_API_KEY` | ŌĆö | Same for mistral |
| `OPENROUTER_API_KEY` | ŌĆö | Same for openrouter |
| `COHERE_API_KEY` | ŌĆö | Same for cohere |
| `HF_API_KEY` | ŌĆö | Same for huggingface |

**Provider env vars are a convenience**, not a permanent source. They're only
read on startup when seeding defaults. After that, keys live in the DB and the
admin UI is the source of truth. If you rotate a key via env var after the
initial seed, the new value is ignored.

## 3. Observability

### 3.1 Structured logs

Every log line is JSON (when `FREEAI_LOG_FORMAT=json`) with a shared set of
fields:

```json
{
  "event": "completed",
  "level": "info",
  "logger": "freeai.orchestrator",
  "timestamp": "2026-04-11T19:42:00",
  "request_id": "a1b2c3d4e5f6g7h8",
  "method": "POST",
  "path": "/v1/chat/completions",
  "provider": "groq",
  "strategy": "coding",
  "latency_ms": 487,
  "tokens": 320,
  "chain": ["groq"]
}
```

The `request_id` is the glue: it's attached to every log line inside a request
thanks to `structlog` contextvars, and echoed back as `X-Request-ID` in the
response. A client that logs an error can ship you the request_id and you
`grep request_id=ŌĆ” logs.json` to get the whole server-side story.

Key events you'll see:

| `event` | Level | When |
|---|---|---|
| `freeai_ready` | info | Startup, after migrations and seeding |
| `bootstrap_mode` | warn | Startup, if no clients exist and auth isn't required |
| `seeded_default_providers` | info | First run, per provider seeded |
| `seeded_builtin_strategies` | info | First run, per strategy seeded |
| `dispatching` | info | Before each provider call, with `provider`, `strategy`, `attempt` |
| `retrying transient error` | info | Before an in-place retry |
| `auto_strategy` | info | When `strategy=auto` is resolved ŌĆö includes detected `language` and all signals |
| `provider over capacity, skipping` | info | When `try_reserve` returns None |
| `completed` | info | Successful completion |
| `provider failed` | warn | One provider failed; includes `kind` and `message` |
| `initial_setup_completed` | info | First-run UI setup saved admin hash + optional provider keys |
| `unhandled_exception` | error | Middleware caught anything not a `ProviderError` |

### 3.2 Prometheus metrics

Exposed at `GET /metrics` (public, no auth). Metric inventory:

**HTTP surface**
- `freeai_http_requests_total{method, path, status}` ŌĆö counter
- `freeai_http_request_duration_seconds{method, path}` ŌĆö histogram, buckets
  `(0.01, 0.05, 0.1, 0.25, 0.5, 1, 2, 5, 10, 30)`

**Provider dispatch**
- `freeai_provider_calls_total{provider, outcome}` ŌĆö counter. Outcomes:
  `success`, `server_error`, `rate_limited`, `auth`, `network`,
  `client_error`, `parsing`, `empty_response`, `content_filtered`,
  `unknown`.
- `freeai_provider_call_duration_seconds{provider}` ŌĆö histogram, buckets
  `(0.05, 0.1, 0.25, 0.5, 1, 2, 5, 10, 30, 60)`

**Orchestrator**
- `freeai_orchestrator_fallbacks_total{from_provider, to_provider}` ŌĆö counter
- `freeai_provider_circuit_breaker_trips_total{provider}` ŌĆö counter. One
  tick each time the breaker flips `healthy=false` for some user of that
  provider (see [ARCHITECTURE.md ┬¦ 4 Circuit breaker](ARCHITECTURE.md#circuit-breaker-and-retry-budget)).
  A sudden rate of trips is a strong signal that an upstream is degraded
  for a meaningful chunk of your users.

**Cardinality warning**: `freeai_http_requests_total` currently uses the raw
`request.url.path`, so `/api/providers/groq` and `/api/providers/gemini`
produce different series. If you add user-controlled paths, cardinality will
explode. See [REVIEW.md ┬¦ 5](REVIEW.md#5-metrics-cardinality).

### 3.3 Grafana dashboard

Available only under the `observability` profile.

Panels:

1. HTTP request rate by status
2. HTTP latency p50/p95/p99
3. Provider calls by outcome (stacked)
4. Provider calls by provider (stacked)
5. Provider latency p95 per provider
6. Fallback transitions (`{from_provider ŌåÆ to_provider}`)
7. Success rate ŌĆö stat panel (stoplight thresholds: red < 90%, orange < 98%, green Ōēź 98%)
8. Total calls (last hour) ŌĆö stat panel
9. Fallback rate ŌĆö stat panel
10. Distinct providers seen ŌĆö stat panel

To add your own panels, edit the JSON and `docker compose restart grafana` ŌĆö
the provisioner re-applies on startup. Or edit in-UI and export.

### 3.4 In-app analytics

The frontend has an **Analytics** tab that reads `/api/analytics`. It's
independent from Prometheus and covers a longer window (the last 7 days
max) from `usage_events`. Use it when:

- You don't have Prometheus set up
- You need per-client or per-strategy breakdowns (Prometheus doesn't have
  those labels)
- You want the history even after restarts (Prometheus retention is its own
  concern)

## 4. Backup and restore

### 4.1 What to back up

| Table | Critical? | Why |
|---|---|---|
| `providers` | **yes** | Has the encrypted API keys |
| `clients` | **yes** | Losing it breaks all inbound auth |
| `strategies` | yes | Custom ones are your config |
| `app_config` | yes | Default strategy + fallback flag |
| `provider_stats` | no | Regenerated on first use |
| `rate_events` | no | Losing it means the first minute looks cold |
| `usage_events` | depends | Historical analytics ŌĆö back up if you report on it |

Also back up, **outside the database**:

- **`FREEAI_MASTER_KEY`** ŌĆö without it, the encrypted provider keys are
  useless. The same applies to `data/.master_key` if you're letting the app
  autogenerate. **This is the single most important secret.**
- `FREEAI_ADMIN_TOKEN` ŌĆö mild; you can rotate it any time.

### 4.2 Backup script

```bash
#!/usr/bin/env bash
set -euo pipefail
STAMP=$(date +%Y%m%d-%H%M%S)
OUT=/backups

# Config tables (small, back up daily)
pg_dump -U freeai -d freeai \
  --table=providers --table=clients \
  --table=strategies --table=app_config \
  --data-only --format=custom \
  -f "$OUT/freeai-config-$STAMP.dump"

# Usage events (big, back up less often if you care)
pg_dump -U freeai -d freeai \
  --table=usage_events \
  --data-only --format=custom \
  -f "$OUT/freeai-usage-$STAMP.dump"

# Master key ŌĆö plain cp, chmod 600
install -m 600 /app/backend/data/.master_key "$OUT/master-key-$STAMP"
```

### 4.3 Restore

Schema first (from Alembic), then data:

```bash
# Fresh database ŌĆö run migrations
FREEAI_DATABASE_URL=... alembic upgrade head

# Restore config + usage
pg_restore -U freeai -d freeai --data-only freeai-config-*.dump
pg_restore -U freeai -d freeai --data-only freeai-usage-*.dump

# Put the master key back
cp master-key-* /app/backend/data/.master_key
chmod 600 /app/backend/data/.master_key
```

Wrong master key = the provider API keys decrypt to garbage, `/v1/*` calls
fail with `auth` errors (because the provider rejects the mangled bearer
token). If you land in this state with no master key backup, clear
`providers.api_key_encrypted` and re-enter the keys from the UI.

## 5. Running behind a reverse proxy

Typical config: FreeAI bound to `127.0.0.1:8000`, nginx/traefik terminating
TLS in front.

**nginx:**

```nginx
upstream freeai { server 127.0.0.1:8000; }

server {
    listen 443 ssl http2;
    server_name freeai.example.com;

    # TLS config omitted

    location / {
        proxy_pass         http://freeai;
        proxy_http_version 1.1;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_set_header   X-Request-ID      $request_id;

        # Streaming ŌĆö don't buffer SSE
        proxy_buffering off;
        proxy_cache     off;
        proxy_read_timeout 300s;
    }

    # Optionally gate /metrics to internal only
    location = /metrics {
        allow 10.0.0.0/8;
        deny all;
        proxy_pass http://freeai;
    }
}
```

Things to get right:

- **`proxy_buffering off`** on the chat endpoint ŌĆö otherwise nginx holds the
  entire SSE response before flushing to the client, and streaming looks
  completely broken.
- **`X-Forwarded-For` / `X-Forwarded-Proto`** ŌĆö FastAPI doesn't have a
  `ProxyHeaders` middleware enabled by default. If you need the real client IP
  in logs, add one or trust the header yourself.
- **Expose `/metrics` only internally** ŌĆö no auth. Anyone with the URL can
  read counts and latencies. Not a security disaster (no secrets) but it's
  basic hygiene.

## 6. Troubleshooting

### "admin auth required" on every admin request

- Check you're sending `X-Admin-Token: adm_ŌĆ”` or
  `Authorization: Bearer adm_ŌĆ”`.
- If running in Docker and the token was autogenerated, grab it:
  `docker compose exec freeai cat /app/backend/data/admin_token`.
- If you lost it, delete the file and restart ŌĆö a new one will be generated.

### All providers show "no key" in the UI

The seed job seeds the `providers` table but leaves `api_key_encrypted` NULL.
Either:

- Paste keys in the admin UI
- Set env vars **before the first startup** ŌĆö `GROQ_API_KEY`, etc. They're
  picked up at seed time only.

### Everything works but streams arrive in one big chunk

Usually a proxy buffering issue. Check:

- nginx: `proxy_buffering off`
- cloudflare: streaming is only supported on paid plans for non-HTML responses
- traefik: `buffering` middleware not active on this route

Verify by hitting FreeAI directly bypassing the proxy ŌĆö if the stream is fine
there, the proxy is the problem.

### "no provider configured/available" on every request

- `GET /api/providers` ŌĆö are any enabled and `has_key: true`?
- `GET /api/providers` ŌĆö is any provider in `healthy: false`? If all are
  quarantined, nothing ranks high enough. Run
  `POST /api/providers/{name}/reset` to clear.
- `SELECT * FROM provider_stats;` ŌĆö look at `quarantined_until` and
  `consecutive_failures`.

### Slow startup

- `FREEAI_AUTO_MIGRATE=true` + a fresh database ŌĆö migrations take a few
  seconds. Subsequent starts are instant.
- Check logs for `running_migrations` / `migrations_done` bracket.

### Request rate limit mystery ("503 but I'm barely using it")

Possible causes:

1. **Per-client rate limit hit** ŌĆö your client key has an `rpm_limit` set
   low. Check in the UI or `SELECT * FROM clients`.
2. **Provider rate limit hit** ŌĆö the upstream 429'd us and we're waiting
   for `Retry-After`. Check `provider_stats.quarantined_until`.
3. **All providers exhausted their rpd** ŌĆö check `rate_events` with the
   query in [DATABASE.md ┬¦ 6](DATABASE.md#6-connecting-for-ad-hoc-queries).

### Circuit breaker tripped too aggressively (or not enough)

Symptom: a healthy provider keeps landing in quarantine, or a flaky one
still gets picked every time.

Check the current state for a given user:

```sql
SELECT provider_name, healthy, consecutive_failures,
       to_timestamp(quarantined_until) AS quar_until,
       cooldown_level, last_error_kind, substr(last_error,1,80) AS err
FROM provider_stats WHERE user_id = 1 ORDER BY provider_name;
```

If `consecutive_failures` climbs fast for one provider, the upstream is
genuinely degraded. If it climbs fast across **all** providers, the
problem is likely client-side (bad payloads, network between client and
FreeAI). Tune `app_config.circuit_breaker_*` columns ŌĆö see
[API.md ┬¦ Fallback robustness tunables](API.md#fallback-robustness-tunables-app_config).

To manually clear a stuck quarantine on one provider:

```sql
UPDATE provider_stats
SET healthy=true, consecutive_failures=0, quarantined_until=0,
    cooldown_level=0, recent_failures_started_at=0
WHERE user_id=1 AND provider_name='groq';
```

The Prometheus counter `freeai_provider_circuit_breaker_trips_total{provider}`
is the best early-warning signal ŌĆö an alert at `rate(trips_total[5m]) > 0.5`
catches most real incidents.

### Streams that feel stuck mid-response

FreeAI adds an **idle timeout** to streaming upstreams (default 45s without
a chunk). If an upstream accepts the request but never sends bytes, the
orchestrator raises `NETWORK` and falls back ŌĆö **unless** bytes were
already flushed, in which case the stream is aborted with the client. Tune
via `app_config.stream_idle_timeout_s`. A value too low (e.g. 10s) will
cut off legitimately slow responses from reasoning models; too high
(>2min) lets a dead upstream hold the client forever.

### Rate counters wrong after a burst of client retries

Symptom: `provider_stats.consecutive_failures` keeps growing even though
no actual provider call fails; `rate_events` has many rows from the same
user but `usage_events` shows very few.

This was a bug before Sprint 7 ŌĆö cancelled requests (client timed out and
gave up) left rate_events reservations uncommitted. The fix (commits
`08e2f0f` and `8e789f9`) wraps every endpoint in `try/finally` that
rolls back the reservation on cancel. If you see this on Sprint 7+ code,
report the exact repro.

### Postgres connection pool exhausted

Symptom: requests hang for ~30s then 504. Cause: pool size ├Ś workers
exceeds Postgres `max_connections`, or a slow query is blocking a connection.

Fixes:

- Lower `FREEAI_DB_POOL_SIZE` ├Ś worker count to fit within Postgres `max_connections`
- Check for long-running queries: `SELECT pid, query, now() - query_start FROM pg_stat_activity WHERE state = 'active' ORDER BY query_start;`
- Usually it's the rate tracker under load ŌĆö batch the snapshot calls in `_rank`
  ([REVIEW.md ┬¦ 2](REVIEW.md#2-hot-path-inefficiencies))

## 7. Upgrading

1. Pull the new image / code
2. Read the changelog and any new migrations in `alembic/versions/`
3. If you set `FREEAI_AUTO_MIGRATE=true`, starting the new version runs the
   migrations automatically
4. If `auto_migrate=false`, run `alembic upgrade head` before starting traffic
5. Watch the logs for `freeai_ready` and `migrations_done`
6. Sanity-check the app is serving: `curl http://localhost:8000/api/health`
   should return `{"status":"ok"}`

Rollback: migrations have `downgrade()` but we haven't exercised them.
**If you're rolling back across a migration boundary, restore the backup
instead** ŌĆö safer than trusting untested downgrades.
